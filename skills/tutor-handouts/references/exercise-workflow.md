# Exercise Workflow ‚Äî E1‚ÄìE4 Detail

Companion to `handout-workflow.md` TH4 and TH5. Use when emitting Programming Exercises (E1) and the autograder harness (E3).

## Naming Convention

- Exercise slug: lowercase-hyphen, max 32 chars (e.g., `vector-add`, `tiled-matmul`, `cp-async-pipeline`)
- Folder: `Coursepack/Exercises/{NN-topic}/{NN-slug}/` where NN is the two-digit index within the topic
- Bank ID: `E-{STACK}-{NN}` where STACK is `CUDA`, `TRI` (Triton). Used in cross-references between handouts and exercises.

### `challenge.py` (autograder spec)

Each `challenge.py` is a self-contained Python file that:
1. Subclasses `ChallengeBase` from `../../_harness/challenge_base`
2. Sets `metadata: ChallengeMetadata` with name / topic / difficulty / atol / rtol / target_frameworks
3. Implements `reference_impl(*args)` ‚Äî pure torch, ‚â§30 lines
4. Implements `generate_example_test()` ‚Äî 1 tiny case for quick `make test`
5. Implements `generate_functional_test()` ‚Äî 3‚Äì6 cases of varying size
6. Implements `generate_performance_test()` ‚Äî 1 large case for `make bench`

Template at `exercise-templates/challenge.template.py`.

### `starter/starter.{cu,triton.py,pytorch.py}` ‚Äî required properties

| Property | Test |
|---|---|
| Compiles / imports cleanly | `nvcc -E starter.cu` exits 0; `python -c "import sys; sys.path.insert(0, 'starter'); import starter"` exits 0 |
| Does NOT solve | All FIXME markers leave logic incomplete; running starter prints values that fail correctness |
| Has ‚â•3 numbered FIXMEs | `grep -c '^[[:space:]]*//[[:space:]]*FIXME' starter.cu` ‚â• 3 |
| FIXMEs cite vault concepts | Each FIXME line contains `(see StudyVault/...)` |
| Includes a working `main()` test harness | A learner can `make starter && ./build/starter` and see (wrong) output |

### `reference/reference.{cu,triton.py,pytorch.py}` ‚Äî required properties

| Property | Test |
|---|---|
| Compiles + runs cleanly | `make reference` succeeds |
| Output `torch.allclose` with `challenge.py::reference_impl()` | Validated by `runner.py --validate-reference` |
| No FIXME markers | `grep -c FIXME reference.cu` == 0 |
| Same I/O signature as starter | Both use the same `main()` API |

### `Solutions/` (‚òÖ‚òÖ‚òÖ only)

Identical to `reference/` plus `SOLUTION_KEY.md`:

```markdown
# {Exercise Title} ‚Äî Solution Key

## Key Insight
{1 paragraph on the algorithmic idea ‚Äî e.g., "The cp.async pipeline overlaps global-to-shared
copy of tile N+1 with the compute on tile N. The two-stage buffer A[0]/A[1] is rotated each
iteration; cp.async.commit_group + cp.async.wait_group(1) creates the pipeline depth."}

## Common Student Mistakes
1. **Forgetting `cp.async.commit_group` between issues** ‚Äî symptom: hangs or random data; fix: every batch of `cp.async` issues must be followed by `commit_group`.
2. **Wrong `wait_group` depth** ‚Äî symptom: race condition on first iteration; fix: depth = pipeline stages - 1.
3. **Bank conflicts in the shared tile** ‚Äî symptom: 3-4x slowdown; fix: pad the inner dimension by 4 or apply XOR swizzle.

## Canonical Reference
- CUDA C++ Programming Guide ¬ßB.27 (cp.async)
- CUTLASS `include/cute/atom/copy_atom.hpp` ‚Äî `Copy_Atom<SM80_CP_ASYNC_*>` family
```

### `Makefile`

Single template, parameterized via bank entry's `Performance target` field for `ARCH`. From `exercise-templates/Makefile.template`:

```makefile
# Generated by tutor-handouts.
NVCC ?= nvcc
ARCH ?= -arch=sm_80
CXXFLAGS ?= -O3 -std=c++17 -lineinfo
PYTHON ?= python

BUILD := build

all: starter

$(BUILD):
	@mkdir -p $(BUILD)

starter: $(BUILD) starter/starter.cu
	$(NVCC) $(ARCH) $(CXXFLAGS) -o $(BUILD)/starter starter/starter.cu

reference: $(BUILD) reference/reference.cu
	$(NVCC) $(ARCH) $(CXXFLAGS) -o $(BUILD)/reference reference/reference.cu

test:
	$(PYTHON) ../../_harness/runner.py $(CURDIR)

bench:
	$(PYTHON) ../../_harness/runner.py $(CURDIR) --bench

clean:
	rm -rf $(BUILD)/

.PHONY: all starter reference test bench clean
```

## E3 ‚Äî Harness Anatomy

### `_harness/challenge_base.py` (verbatim from skill bundle)

Provides:
- `ChallengeMetadata` dataclass
- `ChallengeBase` abstract class
- `TestReport` and `BenchReport` dataclasses
- `validate(candidate, framework)` ‚Äî runs candidate against example+functional tests
- `benchmark(candidate, framework)` ‚Äî runs against performance test

### `_harness/runner.py` (verbatim from skill bundle)

CLI:
```
runner.py [<exercise_dir>] [--framework cuda|triton|pytorch] [--bench] [--validate-reference]
```

Default behavior (no args): walks `..` for every `challenge.py`, imports, runs `generate_example_test()`. Pure import test by default ‚Äî no CUDA required.

With `--bench`: runs `generate_performance_test()` and measures via `torch.cuda.Event`.

With `--validate-reference`: imports each `reference/reference.{framework}.py`, runs it against `challenge.py::reference_impl()` via `torch.allclose(..., atol=metadata.atol, rtol=metadata.rtol)`. Reports pass/fail per exercise.

## Idempotence

Every emitted file starts with a header:

```
# Generated by tutor-handouts at {ISO8601} ‚Äî re-running the skill overwrites this file.
```

(LaTeX uses `%`, Python uses `#`, Makefile uses `#`, Markdown uses HTML comment `<!-- ... -->`.)

Re-running the skill:
1. Reads each existing file's first-line timestamp.
2. If user has hand-edited (detected via `mtime` newer than the timestamp by > 1 minute), prompt before overwrite.
3. `Solutions/SOLUTION_KEY.md` is ALWAYS prompted before overwrite ‚Äî instructors customize this most.

## Anti-AI-slop for exercises

Per `content-rules/exercise-content-rules.md`:
- No "various" / "several" / "many" in FIXME comments
- Every FIXME points at a specific concept note (relative path required)
- Every reference_impl uses real torch APIs, never pseudocode
- atol/rtol must match the math: FP32 GEMM = 1e-3/1e-4, FP16 = 1e-2/1e-3, integer = 0/0
- Test sizes must be realistic for the kernel: vector-add minimum 1<<22 to amortize launch; matmul minimum 1024√ó1024 to exercise shared memory tiling

## Failure modes to handle gracefully

| Condition | Action |
|---|---|
| `nvcc` not installed | Emit starter/reference anyway. Document in `README.md`: "Requires CUDA Toolkit ‚â• 11.0". |
| `triton` not installed | Same. Document: "Requires Triton ‚â• 2.0". |
| Exercise bank entry references a vault concept that doesn't exist | Skip the FIXME citation, log a warning, do NOT skip the exercise. |
| Two bank entries map to the same slug | Append numeric suffix (`vector-add`, `vector-add-2`). Log a warning. |
| `--frameworks` excludes all frameworks the bank entry supports | Skip the exercise entirely. Log to `Coursepack/README.md`. |
